# Сеть WATERS Node

> P2P-топология, NAT traversal, мастер-слейв, relay, DTN для прерывистой связи.
> Как соединить 6 нод вокруг Земли.

---

## 1. Топология

### 1.1 Прямое P2P (обе ноды с публичным IP)

```
┌─────────┐       P2P TCP       ┌─────────┐
│ Нода A  │ ◄───────────────── ► │ Нода B  │
│ Мск     │                     │ Лондон  │
│ stat IP │                     │ stat IP │
└─────────┘                     └─────────┘
```

- mDNS для локального обнаружения
- TCP sync для WAL-каналов
- Каждая нода: HTTP API + MCP Server + TCP sync + mDNS

### 1.2 Master-Slave (одна нода за NAT)

```
┌──────────┐     TCP (slave→master)     ┌──────────┐
│ Нода A   │ ──────────────────────────► │ Нода B   │
│ Дача     │                             │ VPS      │
│ за NAT   │                             │ stat IP  │
│ динамич. │                             └──────────┘
└──────────┘
```

- Slave сам инициирует соединение к мастеру
- Мастер видит всех слейвов
- Слейвы общаются через мастера (relay)

### 1.3 Hub-and-Spoke (все за NAT)

```
┌──────────┐
│  VPS     │  ← статический IP, мастер
│  (хаб)   │
└────┬─────┘
     │
┌────┴─────┐  ┌────┴─────┐  ┌────┴─────┐
│  Дача    │  │  Дом     │  │  Офис    │
│  NAT     │  │  NAT     │  │  NAT     │
└──────────┘  └──────────┘  └──────────┘
```

- Все клиенты подключаются к VPS
- VPS ретранслирует сообщения между клиентами
- Режим по умолчанию для удалённых команд

### 1.4 Полная сетка (6 нод)

```
     ┌─────────┐
     │  VPS    │
     │  Deep   │
     └────┬────┘
          │
    ┌─────┼─────┐
    │     │     │
┌───┴──┐┌──┴───┐┌───┴──┐
│Дача  ││Дом   ││Лондон│
│Claude││Ollama││GPT-4 │
└──────┘└──────┘└──────┘
```

- 6 нод, каждая со своим LLM
- Каждая нода соединена с каждой (mesh)
- DTN для прерывистых каналов

## 2. Порты

| Сервис | Порт | Относительно HTTP |
|--------|------|-------------------|
| HTTP (веб-дашборд) | N | — |
| MCP Server | N+100 | N + 100 |
| TCP sync | N+3 | N + 3 |
| mDNS | N+1 | N + 1 |

Пример: `--port 42069` →
- Дашборд: 42069
- MCP: 42169
- TCP sync: 42072
- mDNS: 42070

## 3. Протоколы соединения

| Ситуация | Механизм |
|----------|---------|
| Оба публичные IP | Прямой P2P TCP |
| Один за NAT | Master-Slave (slave → master) |
| Оба за NAT | Relay через публичную ноду |
| Локальная сеть | mDNS автообнаружение |
| Нет IP (поле) | DTN-задержки (tc-netem) |

## 4. WAL-синхронизация

```rust
pub enum WalEntry {
    ChatMessage { group_id, text, sender },
    TaskCreated { task_id, title, desc },
    TaskAssigned { task_id, agent_id },
    AgentSpawned { agent_id, role, llm },
    AgentFinding { agent_id, data },
    StatusUpdate { node_id, bridges, load },
}
```

- Каждая запись = seq + timestamp
- Sync: запрос с last_seq → diff только новых записей
- Разрешение конфликтов: приоритет у ноды с static IP
- После sync обе ноды имеют одинаковые seq

## 5. DTN (Delay/Disruption Tolerant Networking)

Для туризма, полевых условий, прерывистой связи:

```
┌─────────┐     обрыв связи     ┌─────────┐
│ Нода A  │ ──── x x x x ──── ► │ Нода B  │
│ горы    │                     │ базовый │
│ L3      │                     │ лагерь  │
└─────────┘                     └─────────┘

1. Связь есть → DTN-буфер → передача сжатых данных
2. Связь прервана → данные в offline queue
3. Связь восстановлена → досыл с последнего seq
4. Приоритет: Cargo > мнения > чат > логи
```

### DTN в цифрах

| Параметр | Значение |
|----------|---------|
| Размер пакета | 256 байт (через LoRa/SMS) |
| Макс очередь | 1000 событий |
| TTL события | 24 часа |
| Сжатие | gzip level 6 |
| Retry | каждые 5 минут, макс 3 |

## 6. Групповой чат между нодами

```
/say 1 привет всем
# Все 6 нод видят сообщение в Redis Stream

/chat 1 как слышно?
# Чат с показом последних сообщений

/opinions 1
# Мнения агентов группы о текущей задаче
```

- Stream: `group:{id}:chat`
- Каждая нода подписана на Stream
- Новые сообщения → SSE → веб-дашборд
